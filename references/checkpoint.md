# 存档与续跑（Checkpoint）

## 为什么必须有这套机制

上下文窗口 = 一次性工作记忆。长调研必须假设它**随时会满 / 被压缩 / 会话中断**（Codex 报 "ran out of context window"、扣子重开窗口、Claude 撞限流——都发生过）。对策只有一个：**文件是唯一可靠的记忆，窗口只是工作台**。

三条铁律：

1. **任何阶段产物一完成，立刻落盘**——落盘之前它不存在。
2. **STATE.md 永远反映最新进度**——每完成一个阶段（或一个维度）就更新。
3. **任何新窗口都能只靠工作目录恢复全部工作**——不依赖聊天历史。

## 工作目录

```
~/.cache/ray-deep-research/{slug}/
├── STATE.md           进度总控（本文件核心）
├── agent-1.md …       各维度完整素材（末尾有 <!-- RESEARCH-COMPLETE --> 完成标记）
├── outline.md         Stage 2 整合大纲
├── verification.md    Stage 2.5 审查 suspect 清单 + 逐项核查结论
└── report.html        Stage 3 起的报告文件（分章节写入，从半成品到成品）
```

**为什么不再用 /tmp**：mac 重启即清空，部分沙箱环境每次会话都重置——断点存了等于白存。`~/.cache` 跨重启、跨会话存活。（旧版用 `/tmp/ray-research/`；若在那里发现历史存档，迁移过来再继续即可。）

## STATE.md 模板

```markdown
# 调研进度 · {主题}
- slug: {slug}
- 类型: investment | competitor | generic
- 参数: {--agents 5 / --quick / --no-publish 等，无则写"无"}
- 模式: full | degraded
- Agent 数: {N}（并行 | 单窗口顺序）
- 开始: {YYYY-MM-DD HH:MM}

## 阶段状态
- [x] Stage 0 主题解析
- [x] Stage 0.5 复用检查（结果：无命中 / 命中已问用户）
- [ ] Stage 1 调研（agent-1 ✅ / agent-2 ✅ / agent-3 ⏳ / agent-4 ⬜）
- [ ] Stage 2 整合大纲
- [ ] Stage 2.5 数据验证
- [ ] Stage 3 渲染 HTML（已写入章节：无）
- [ ] Stage 4 自检
- [ ] Stage 5 交付（入库 / 发布）

## 下一步
{一句话：接下来该干的最小动作，如"跑 agent-3 社区口碑维度"}

## 恢复方法（给下一个窗口的自己）
新窗口收到"继续调研 {主题}"时：读本文件 → 找上面第一个未完成项 → 读对应落盘文件 → 从那里继续。已打勾的阶段绝不重跑。
```

## 恢复流程（最高优先级，先于一切阶段）

**触发条件**（满足其一）：

- 用户说"继续调研 X" / "继续" / "接着跑"
- Stage 0 解析出 slug 后，发现 `~/.cache/ray-deep-research/{slug}/STATE.md` 已存在且未标 DONE

**动作**：读 STATE.md → 找第一个未完成阶段 → 读该阶段依赖的落盘文件（不是聊天历史）→ 用一句话告知用户恢复点（"上次进行到 Stage 2.5，已完成 4 个维度调研，从数据验证继续"）→ 继续跑。**已打勾的阶段绝不重跑**。Stage 1 内部按维度粒度续跑（完成标记规则见 `agent-orchestration.md` 断点续跑节）。

## 各阶段存档点

| 阶段完成时 | 必须落盘 | STATE.md 更新 |
|-----------|---------|--------------|
| Stage 0 / 0.5 | 建工作目录 + 初版 STATE.md | 勾选 + 记录判定结果 |
| Stage 1 每个维度 | agent-N.md（含完成标记） | 该维度打 ✅ |
| Stage 2 | outline.md | 勾选 |
| Stage 2.5 | verification.md（suspect 清单 + 每条核查结论） | 勾选 |
| Stage 3 每写完一章 | report.html 追加该章 | 更新"已写入章节"列表 |
| Stage 4 | 修补直接改 report.html | 勾选 |
| Stage 5 | 报告已入库/发布 | 全部勾选 + 顶部标 `DONE` |

## 单窗口环境的"边读边炼"纪律

无法并行子 Agent 的环境（Codex、扣子等）里，所有原始材料都流经主窗口——这是爆窗的头号原因。并行模式靠"子 Agent 各有各的窗口"天然隔离；单窗口模式必须用纪律补上这道墙：

1. **一次只做一个维度**，做完立刻落盘 + 更新 STATE.md，再开下一个。
2. **每读完 1-2 个来源，立刻把要点（关键数字、结论、来源 URL）写进当前维度的 agent-N.md**。不要连读十个页面攒到最后一起总结——那样十个页面的原文同时压在窗口里。
3. **读过即弃**：要点入档后，后续工作只依赖笔记文件，不回看窗口里的原文。
4. **觉察上下文吃紧就主动换班**（平台弹出容量警告 / 对话已经很长 / 开始记不清早前细节）：立刻把当前半成品落盘 + 更新 STATE.md，然后停下，对用户说换窗话术（见下）。**主动换班是计划内操作，硬撑到爆才是事故。**
5. **Stage 3 写 HTML 分章节落盘**：先复制骨架到 report.html，然后一章一章分次写入文件，每写完一章更新 STATE.md 的章节列表。绝不在窗口里一口气生成整份 2000 行 HTML。

## 换窗话术（吃紧时对用户说）

> 上下文快满了，我先把进度存档（已完成 {X}/{Y}，全部落盘）。请开一个**新窗口**对我说"**继续调研 {主题}**"，我会从断点接着跑，已完成的部分不会重跑。

## 交付后

Stage 5 完成 → STATE.md 顶部第一行加 `> DONE {日期}`。工作目录保留（它就是这次调研的素材档案，复查数据来源时有用）；用户想清理时删 `~/.cache/ray-deep-research/` 下的旧目录即可，skill 不自动删。
